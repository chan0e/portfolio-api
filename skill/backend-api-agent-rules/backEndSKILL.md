# Backend Agent Rules

## 핵심 원칙
- 계약 우선으로 진행하라: 구현 전에 API 명세와 도메인 규칙을 먼저 확정하라.
- 하위 호환을 기본값으로 두어라: 기존 클라이언트가 깨지는 변경을 금지하라.
- 운영 가능성을 코드와 함께 설계하라: 로그, 메트릭, 장애 대응 기준을 동시 정의하라.
- 보안을 후행 작업으로 미루지 말라: 인증/인가/비밀관리/암호화 기준을 초기 설계에 포함하라.

## 표준 작업 순서
1. 요구사항과 성공 조건을 확정하라.
2. API 계약(요청/응답/에러/상태코드)을 문서화하라.
3. 데이터 모델과 마이그레이션 전략을 확정하라.
4. 애플리케이션 계층(Controller/Service/Repository) 책임을 분리하라.
5. 예외 처리, 검증, 에러 포맷을 표준화하라.
6. 테스트(단위/통합/스모크)와 품질 게이트를 작성하라.
7. 운영 관측성(로그/메트릭/트레이싱)과 배포 체크리스트를 점검하라.

## API 규칙
- 리소스 중심 경로를 사용하라. 예: `/api/v1/portfolio`.
- 성공 응답과 에러 응답의 콘텐츠 타입을 명시하라.
- 비-200 응답 정책을 누락하지 말라: `304/400/401/403/404/406/409/422/429/500/503`를 검토하라.
- 에러 포맷은 `application/problem+json`을 기본으로 사용하라.
- `X-Request-Id`를 수용/반환하여 추적 가능성을 보장하라.
- 캐시 가능한 조회 API는 `ETag`와 `If-None-Match`를 정의하라.
- 페이로드에서 `null`/빈 배열 정책을 명확히 하라.

## 데이터/DB 규칙
- 스키마는 API 계약을 충족하도록 설계하라.
- 정렬/노출 제어용 필드(`order`, `is_visible`)를 필요 시 명시하라.
- 마이그레이션은 재실행 가능하고 롤백 전략을 포함하라.
- 인덱스는 조회 패턴 기준으로 설계하고 근거를 남겨라.
- 문자열 기간 값 등 UI 의존 필드는 별도 정책을 문서화하라.

## 보안/암호화 규칙
- 전송 구간은 TLS(HTTPS) 강제, HSTS 적용을 기본으로 하라.
- 저장 구간은 디스크/백업 암호화를 기본으로 하라.
- 민감 정보는 필드 단위 암호화(AES-256-GCM) 또는 해시 보조 컬럼 전략을 정의하라.
- 비밀값은 코드/저장소에 저장하지 말고 KMS/Vault/Secret Manager로 관리하라.
- 로그에는 토큰/비밀번호/개인정보를 기록하지 말고 마스킹하라.

## 테스트/품질 규칙
- 단위 테스트: 비즈니스 규칙/매핑/검증 로직을 검증하라.
- 통합 테스트: API-DB 연동, 직렬화, 에러 응답 계약을 검증하라.
- 스모크 테스트: 기동/헬스체크/핵심 조회 API를 검증하라.
- 머지 전 최소 게이트: 빌드 성공, 핵심 테스트 통과, 명세-구현 일치 확인.

## 관측성/운영 규칙
- 구조화 로그(JSON)를 사용하고 요청 단위 상관관계를 유지하라.
- 핵심 메트릭(P95/P99 지연, 에러율, 처리량)을 정의하라.
- 장애 응답은 사용자 메시지와 내부 진단 정보를 분리하라.
- 레이트 리밋이 있으면 `429`와 `Retry-After`를 일관되게 제공하라.

## JPA 및 계층 구조 설계 원칙

### 1. 역할 분리
- `Controller`는 HTTP 요청/응답, 상태코드, 인증 정보 전달만 담당하라. 웹 계층 책임을 줄여야 테스트가 단순해지고, 비즈니스 규칙이 HTTP 구현에 묶이지 않는다.
- `Service`는 유스케이스, 트랜잭션, 도메인 규칙 조합의 중심으로 두어라. 조회-검증-변경-저장을 한곳에서 끝내야 트랜잭션 경계와 규칙 적용 위치가 명확해진다.
- `Repository`는 JPA 기반 저장과 Aggregate 단위 조회/변경에 사용하라. 쓰기 모델과 단순 조회는 엔티티 중심 접근이 가장 일관되고 유지보수하기 쉽다.
- `DAO` 또는 Query 전용 컴포넌트는 복잡한 목록 조회, 통계, 검색, 대시보드, 네이티브 SQL 최적화에 사용하라. 이런 영역은 엔티티 그래프보다 SQL 제어력과 응답 형태 최적화가 더 중요하다.

### 2. DTO와 Entity 사용 기준
- DTO를 API 경계의 기본값으로 사용하라. 요청은 Request DTO, 응답은 Response DTO로 분리하라. 엔티티를 외부 계약에 노출하면 필드 변경이 API 변경으로 이어지고, 지연 로딩과 순환 참조 문제도 같이 들어온다.
- DTO는 화면과 유스케이스 기준으로 설계하라. 엔티티 구조를 복사하지 말고, 호출자가 실제로 필요한 값만 담아라. 유스케이스별 DTO가 응답 크기와 변경 영향 범위를 함께 줄인다.
- Entity는 영속성 모델로 한정하라. DB 매핑, 도메인 상태, 핵심 규칙만 두고 요청 파싱이나 응답 포맷 책임은 넣지 말라. Entity가 너무 많은 책임을 가지면 JPA 제약이 시스템 전역으로 퍼진다.
- 조회 전용 응답은 DTO/Projection으로 받고, 엔티티는 변경이 필요한 유스케이스에 집중시켜라. 최근 실무는 "엔티티를 많이 재사용하는 구조"보다 "읽기 모델과 쓰기 모델을 분리하는 구조"가 성능과 유지보수에 더 유리하다.

### 3. 엔티티 코드 작성 원칙

#### 3-1. 기본 구조
- 엔티티는 `field access`를 기본값으로 고려하라. 매핑 어노테이션을 필드에 두고, JPA가 필드를 직접 읽고 쓰게 하라.
  getter/setter를 영속성 요구사항 때문에 억지로 열어둘 필요가 없고, `@Version` 같은 필드도 외부에 숨길 수 있다.
- 엔티티 클래스와 영속 필드는 `final`로 만들지 말고, `public` 또는 `protected` 기본 생성자를 제공하라.
  Jakarta Persistence는 no-arg 생성자와 non-final 제약을 요구하며, 프록시와 지연 로딩도 이 전제 위에서 동작한다.
- 엔티티는 DB 매핑과 도메인 규칙에 집중하라.
  요청 파싱, 응답 포맷, 화면용 계산까지 엔티티에 넣으면 책임이 과도하게 커진다.

#### 3-2. Getter/Setter 원칙
- getter는 필요한 것만 공개하라.
  모든 필드를 열어두면 계층 간 결합이 커지고, DTO 분리도 어려워진다.
- setter는 무분별하게 열지 말라.
  엔티티는 "값 저장용 통"이 아니라, 유효한 상태 변화만 허용하는 도메인 객체여야 한다.
- 특히 `id`, `version`, 생성 시점에 고정되는 값, 연관관계 컬렉션에는 public setter를 두지 말라.
  식별자와 버전, 컬렉션 참조는 외부에서 임의로 바꾸면 영속성 추적과 정합성이 쉽게 깨진다.

#### 3-3. 상태 변경 방식
- 상태 변경은 `setStatus()` 같은 범용 setter보다 `approve()`, `cancel()`, `changeEmail()`, `assignOwner()` 같은 의미 있는 메서드로 표현하라.
  검증, 연관 객체 동기화, 이벤트 처리 지점을 한곳으로 모을 수 있다.
- 생성도 setter 조합보다 정적 팩토리나 의미 있는 생성 메서드로 유도하라.
  예: `Order.create(...)`, `Member.register(...)`
  필수값 누락과 잘못된 초기 상태를 줄일 수 있다.

#### 3-4. 연관관계와 컬렉션
- 연관관계는 기본적으로 단방향 매핑으로 시작하라.
  먼저 쓰기 흐름과 조회 요구를 분석한 뒤, 반대편 탐색이 실제로 필요한 경우에만 양방향 매핑을 추가하라.
- 컬렉션 연관관계는 `null` 대신 빈 컬렉션으로 시작하라.
  `null` 분기와 초기화 누락을 줄일 수 있다.
- 컬렉션 자체를 통째로 교체하는 setter는 두지 말라.
  JPA 컬렉션은 영속성 컨텍스트와 연결되므로 참조를 함부로 바꾸면 추적이 꼬이기 쉽다.
- 양방향 연관관계는 편의 메서드로 양쪽을 함께 맞춰라.
  예: `addItem()`, `removeItem()`
  한쪽만 변경하면 메모리 상태와 DB 반영 의도가 어긋난다.

#### 3-5. 공통 메서드 주의사항
- `equals()`/`hashCode()`는 엔티티에 일괄 생성하지 말라.
  모든 필드 기반 생성이나 단순 ID 기반 생성은 프록시, 영속 상태 전환, 컬렉션 동작과 충돌할 수 있다.
- 필요하다면 안정적인 비즈니스 키가 있을 때만 신중하게 구현하라.
  Hibernate 공식 문서도 이 주제가 단순하지 않다고 설명한다.
- `toString()`에 지연 로딩 연관관계와 컬렉션을 넣지 말라.
  로그 출력 한 번으로 예기치 않은 쿼리, 순환 참조, 성능 저하가 발생할 수 있다.

#### 3-6. 값 객체와 Lombok
- 주소, 금액, 기간처럼 함께 움직이는 값은 `@Embeddable`로 분리하라.
  원시 타입 나열보다 값 객체로 묶는 편이 의미와 제약을 더 잘 캡슐화한다.
- 최신 Jakarta Persistence 3.2는 `record` 기반 embeddable도 허용하지만, 도입 전에는 현재 Hibernate/Spring Boot 버전 호환성을 먼저 확인하라.
  스펙 지원 시점과 실제 런타임 지원 시점이 다를 수 있다.
- 엔티티에 Lombok `@Data`, 전필드 `@Setter`, 전필드 `@EqualsAndHashCode`, 전필드 `@ToString`를 관성적으로 붙이지 말라.
  이런 자동 생성은 지연 로딩, 순환 참조, 잘못된 동등성 비교, 무분별한 상태 변경을 쉽게 만든다.

#### 3-7. 권장 코드 형태
```java
@Entity
public class Order {

    @Id
    @GeneratedValue
    private Long id;

    @Version
    private Long version;

    @Enumerated(EnumType.STRING)
    private OrderStatus status;

    protected Order() {
    }

    private Order(OrderStatus status) {
        this.status = status;
    }

    public static Order create() {
        return new Order(OrderStatus.CREATED);
    }

    public OrderStatus getStatus() {
        return status;
    }

    public void approve() {
        if (status != OrderStatus.CREATED) {
            throw new IllegalStateException("approve not allowed");
        }
        this.status = OrderStatus.APPROVED;
    }

    public void cancel() {
        if (status == OrderStatus.SHIPPED) {
            throw new IllegalStateException("cancel not allowed");
        }
        this.status = OrderStatus.CANCELED;
    }
}
```

### 4. JPA 조회 원칙
- JPA를 모든 조회 문제의 만능 해법으로 쓰지 말라. 쓰기 트랜잭션과 단순 조회에는 JPA를 사용하고, 복잡한 목록/집계/통계 조회는 DTO 프로젝션, JPQL, 네이티브 SQL, 또는 팀 표준 쿼리 도구로 분리하라. 최근 실무는 엔티티 중심보다 유스케이스별 조회 최적화를 더 중시한다.
- 연관관계는 기본적으로 지연 로딩(`LAZY`)으로 시작하라. 특히 `@ManyToOne`, `@OneToOne`도 명시적으로 검토하라. eager 로딩에 기대면 의도하지 않은 추가 쿼리와 과도한 조인이 누적된다.
- 조회 전략은 `Repository`/`DAO` 메서드 수준에서 명시하라. 필요한 시점에만 fetch join 또는 `@EntityGraph`를 사용하라. 누가 언제 무엇을 읽는지 드러나야 N+1과 과조회 문제를 통제할 수 있다.
- `OSIV(Open Session In View)`를 기본값으로 두지 말라. 응답 직렬화 시점의 지연 로딩에 의존하지 말라. 웹 계층으로 쿼리가 새면 성능, 예외 처리, 트랜잭션 일관성이 동시에 흔들린다.

### 5. JPA 쓰기와 트랜잭션 원칙
- 변경 감지는 짧고 명확한 트랜잭션 안에서만 활용하라. 엔티티를 오래 붙잡고 여러 계층을 넘기지 말라. 트랜잭션이 길어질수록 예측 가능성이 떨어지고 락 경합 가능성이 커진다.
- 연관관계는 단방향을 기본값으로 두고, 분석 결과 반대 방향 탐색이 반복적으로 필요할 때만 양방향으로 열어라. 양방향 연관관계와 컬렉션 매핑은 최소화하라. 컬렉션이 많은 엔티티는 flush와 dirty checking 비용이 빠르게 커진다.
- 대량 수정/삭제는 벌크 쿼리로 처리하라. 실행 직후 영속성 컨텍스트를 비우거나 재조회하라. 벌크 연산은 1차 캐시를 건너뛰므로 후속 로직이 오래된 상태를 믿으면 정합성 버그가 생긴다.
- 동시성 제어가 필요한 엔티티에는 `@Version` 기반 낙관적 락을 우선 검토하라. 다중 인스턴스와 비동기 처리 환경에서는 "마지막 저장이 이김" 전략이 데이터 유실로 이어지기 쉽다.
- 배치 성능이 중요한 쓰기 작업은 ID 전략부터 점검하라. 대량 insert가 필요하면 DB 시퀀스와 배치 설정에 맞는 전략을 고르고, `IDENTITY` 사용 시 배치 이점이 줄어드는지 확인하라. JPA 성능은 코드보다 ID 생성 전략과 flush 타이밍 영향을 크게 받는다.

### 6. 운영과 테스트 원칙
- 감사 컬럼(`createdAt`, `updatedAt`, `createdBy`)과 소프트 삭제는 기본값으로 넣지 말라. 운영/법적 요구가 있을 때만 채택하라. 특히 소프트 삭제는 모든 조회 조건과 인덱스 전략을 복잡하게 만든다.
- JPA 테스트는 인메모리 DB만으로 끝내지 말라. 가능하면 Testcontainers 등으로 실제 DB 방언과 제약조건을 검증하라. 인덱스, 락, JSON 컬럼, 함수 차이는 대체 DB에서 재현되지 않는 경우가 많다.

## 산출물 체크리스트
- API 명세 문서가 최신 상태인가
- 상태 코드/에러 코드 정책이 문서와 구현에 일치하는가
- 마이그레이션 스크립트가 신규 환경에서 재현 가능한가
- 보안/암호화/비밀관리 정책이 문서화되었는가
- 테스트 결과와 잔여 리스크가 기록되었는가
