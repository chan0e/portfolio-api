# DB 아키텍처 에이전트 룰스

## 시작 원칙
- 테이블 설계 전에 API 계약과 도메인 모델을 먼저 읽어라.
- 필수/선택 필드, 정렬 규칙, 카디널리티를 먼저 추출하라.
- 예상 조회/쓰기 쿼리를 먼저 정의한 뒤 인덱스를 설계하라.

## 작업 순서
1. 범위, 가정, 성공 기준을 명확히 하라.
2. API 필드를 테이블과 관계로 매핑하라.
3. 컬럼 타입, null 허용, 네이밍 규칙을 결정하라.
4. 비즈니스 불변식을 `PRIMARY KEY`, `FOREIGN KEY`, `UNIQUE`, `CHECK`로 강제하라.
5. 인덱스는 실제 접근 패턴 기반으로만 추가하고 쓰기 비용을 함께 설명하라.
6. 하위 호환, 롤백, 백필 전략을 포함한 마이그레이션 계획을 작성하라.
7. 민감정보에 대해 암호화/해시/키 버전 관리 경계를 정의하라.
8. 대표 쿼리와 엣지 케이스로 설계를 검증하라.

## 기본 산출물
- 카디널리티가 포함된 ERD
- 제약조건 이름이 명시된 DDL 또는 마이그레이션 스크립트
- API 필드 대 테이블 컬럼 매핑표
- 인덱스 전략과 근거
- 남은 리스크와 후속 결정 사항 목록

## 품질 게이트
- API 계약과 DB 제약조건을 항상 동기화하라.
- 정렬/유니크 의미가 깨지는 변경을 금지하라.
- 운영 환경 변경은 가급적 점진적이고 하위 호환적으로 설계하라.
- 마이그레이션/롤백 계획이 없는 제안은 반려하라.
